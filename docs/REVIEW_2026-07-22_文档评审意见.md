## P1 — 提高复刻质量的建议

### P1-1 ⭐ 用"场景解析工具"替代"密度撒点近似"（最大的质量+自动化双赢点）

02 阶段 6.4 / 03 E15 的思路是"按密度图+随机种子撒点还原 ~8000 草 + ~900 树 + ~5300 贴花"。但源场景 YAML 里有**全部 16235 个实例的精确 Transform**，PineForest（10289 个）和 Winter（8087 个）同理。近似撒点是在有精确答案的情况下选择了近似解。

建议：新增 `tools/scene_parser/`（Python）——解析源 `.unity` YAML → 输出 `{prefab名, position, rotation, scale}` JSON → `MyNatureBuilder_Terrain.cs` 读 JSON 摆放。收益：
- 复刻保真度从"分布相似"提高到"逐实例一致"，同机位对比截图直接可比；
- 100% 可自动化、可重跑、可验证（实例数对账）;
- E15 教学点从"随机撒点参数设计"变成"数据驱动还原"（撒点算法仍可作为知识点讲，但产出用精确数据）。
- 注意例外：草若走 Terrain Detail（源项目本身如何实现草需在 05 技术规格中查明——若源场景 6852 株草是 Prefab 实例而非 Terrain Detail，"为什么草用 Terrain Detail"这个 E15 知识点的前提也要复核）。

### P1-2 源项目自带官方 PDF 文档，未被任何文档引用

`Assets/IL3DN/IL3DN_Documentation_v1.7.pdf` 存在。它是顶点色约定、参数含义、prefab 用法的**权威出处**，调研报告目前全靠逆向。建议 01 增加"官方文档"一节，关键约定（顶点色、风/雪参数语义）与 PDF 交叉验证。另外源项目还带 4 个 `Install - IL3DN (URP/HDRP/LWRP).unitypackage`，其中 URP 包可解开对照不同管线下的官方 shader 写法，是 URP 17 迁移卡壳时的额外参考。

### P1-3 顶点色约定可能不完整（40 个模型的返工风险）

01 §4 断言顶点色只用 `.g` 和 `.a`。抽查发现 `IL3DN_Pine.shader` 中存在 `color.rgb` 的使用（Grass 只见 `.g/.a`，与文档一致）。`color.rgb` 也可能是插值后的材质色而非顶点色，**需要逐 shader 确认**。这直接决定 Blender 建模脚本的涂色逻辑——如果某些 shader 用了 `.r` 或 `.b`，40 个模型全部要返工。建议列入 06_模型清单 的前置调研项，并在 05_技术规格中给出"逐 shader 顶点色通道使用表"。

### P1-4 验收标准全部是主观描述，建议量化

各阶段验收是"视觉接近源 Demo""风格近似源项目"。对"自动化 + 视频对比"两个目标来说，建议定义可脚本化的客观指标：
- **同机位截图对比**：源/复刻各出一组固定机位截图（源项目打不开就用 Winter/Demo 场景 YAML 里的相机参数），计算 SSIM / 直方图距离，给出阈值（如 SSIM ≥ 0.75 算通过，逐阶段收紧）；
- **对账类指标**：实例数、材质数、shader 编译 0 error、全局变量名逐一 diff；
- 把这些做成 `MyNatureBuilder_Screenshots.cs` + Python 对比脚本，纳入 02 §4A 的自动化架构（现在 4A.2 表里"视觉校色 ~50%"这行就有了工具支撑，人只看 borderline 案例）。

### P1-5 05/06/07 三份规格文档"实施阶段再写"太晚

03 的 E04（建模）依赖顶点色/尺寸规格，E05（shader）依赖 API 对照表，E10 依赖贴图规格。这三份文档是 **AI Agent 自动化执行的输入**，不是事后记录。建议改为"滚动式提前一集完成"：E03 开工前完成 05 的 shader 部分，E04 前完成 06 的树模型部分，以此类推，并写进每集 checklist。

## P2 — 提高视频质量的建议

### P2-1 ⭐ E10（贴图）排在 E03-E09 之后，前七集画面全是临时贴图

这是大纲最大的观感风险：观众在 E03-E09 整整七集里看到的天空、树、草、岩全部使用"临时贴图"，插画风格的核心卖点（低饱和手绘感）到第 10 集才出现；且如果 E10 之后想重录前面集的素材，成本极高。三选一：
1. **推荐**：拆一个"最小贴图包"前置——在 E03/E04 之间加半集或在 E04 内并入，用 ~5 张核心贴图（bark/leaf/grass/pine/windnoise）先把画面立住，E10 再讲完整生成体系与替换全量；
2. 保持现状，但在 E01 开场和每集片头明确"前期用占位贴图，E10 会有整体翻转"，把它变成刻意的叙事钩子（E10 的前后对比会非常炸）；
3. E10 整体前移到 E05 之前（代价：Python 支柱过早插入，打断 shader 学习曲线）。
任选其一都行，但**必须显式决策**，不能默认。

### P2-2 E11（Water，源 1284 行 5 pass）没有拆集备选

附录 C 给 E07 准备了拆集备选（E07a/E07b），但 E11 同样有超时风险（5 pass + 深度 + 双法线 + 双 Fresnel，20-30 分钟讲不完），备选只有"砍 pass"这个牺牲质量的选项。建议补充"拆 E11a（深度三板斧）/ E11b（法线+Fresnel+泡沫）"备选。

### P2-3 FBX 坐标轴描述易误导（E04 知识点）

03 E04 写"FBX 导出设置（Y 前 Z 上…）"。Blender→Unity 的正确设置是 **Forward = -Z, Up = Y**（或依赖 FBX 默认+Apply Transform）。"Y 前 Z 上"含义不清，照读进视频会教错。建议写成明确的 Blender 导出面板参数（`-Z Forward / Y Up` + Apply Scalings: FBX Units Scale），并在 06_模型清单中固化为导出脚本参数（自动化后人不再手点导出面板，这个知识点讲原理即可）。

### P2-4 外部阻塞项应前置为"启动条件"，不要埋在 04 §8

`voice.wav` 参考音色、AutoVideo 引擎位置（BUILD.md）两项没解决，**第一支视频就出不来**，但它们现在是 04 文档末尾的待办表。建议在 02 §2 状态盘点中列为"视频管线启动条件"，标注"E01 素材录完前必须就绪"，并明确谁提供、验收方式（voice.wav：10-30s、无底噪、采样率要求）。

### P2-5 slug 与目录命名"一致"的措辞不严谨

04 §1 说"`meta.md` 的 slug 与文件夹英文名一致"，但示例文件夹是 `E01_project_skeleton`（下划线）、slug 是 `e01-project-skeleton`（连字符）。机器核对（checklist 第 6 项想自动化的话）会被这句话坑。建议明确转换规则：`slug = 文件夹名小写 + 下划线转连字符`。

## P3 — 提高自动化率的建议

### P3-1 ⭐ B 类"编辑器 UI 操作录屏"可从 ~30% 提到 ~90%：OBS WebSocket

04 §3.4 B 类镜头目前设计为"脚本演示 + 人按 OBS 启停"。OBS 自带 **obs-websocket**（OBS 28+ 内置），Editor 脚本里用 C# WebSocket 客户端（或起个 Python 小服务）发 `StartRecord/StopRecord`，即可把"人按启停"也消掉：演示驱动器开头连 OBS → 开始录制 → 跑操作序列 → 停止录制 → 文件自动落 `clips/`。人从"每条在场 1 分钟"变成"只做成片验收"。这与 02 §4A"触发方式可插拔"的理念完全一致，建议写进 04 §3.4 作为 B 类首选方案（人工按键降为备选）。

### P3-2 自动化架构缺一个"验证层"

02 §4A 有编排层/触发层/业务层，但没有定义**每步跑完后机器怎么知道成功了**。建议补 L0.5 验证约定：
- 每个 Builder 方法返回结构化结果（生成数量、失败清单），batchmode 退出码非 0 即中断编排；
- 编译门禁：每集结束自动跑 `-batchmode -quit`（编译错误即 fail）+ 可选 EditMode 测试；
- 视觉门禁：P1-4 的截图对比脚本;
- 对账门禁：资产计数 vs 05/06/07 规格清单。
有了这层，02 §4A.3"一条 PowerShell 总控脚本顺序调 N 次 batchmode"才真正能无人值守——否则中间一步静默失败，后面全白跑。

### P3-3 每集 checklist（04 §7）本身可以脚本化

9 项 checklist 里至少 5 项可机器检查：clips 分辨率/帧率/像素格式（ffprobe）、meta.md 字段、script.md 每块 `@visual` 路径存在性、块 ID 递增、旁白行长 ≤50 字。建议做一个 `tools/video_lint.py`，构建前跑一遍，把"人自查"降为"人看 lint 报告"。

### P3-4 heightmap / WindNoise 等"数据型资产"的复用边界应明确写出

01 §9 规定"所有贴图必须自建"，但两类资产的性质更接近**数据**而非美术资产：
- `IL3DN_Terrain_Height.raw`（地形形状）：若程序化重造，地形起伏与源不同，所有同机位对比全部失效，撒点坐标（P1-1）也对不上。建议明确"高度图数据允许读取复用（或读取后重采样）"，否则 P1-1 的精确还原自相矛盾；
- `WindNoise`：若噪声图不同，风的相位/频谱与源不同，动态对比视频会有可感差异。程序化生成可以，但文档应写明"允许的差异"。

### P3-5 冬季场景修正后的自动化率表更新

P0-1 修正后，02 §4A.2 表中"场景装配参数还原 ~95%"可以覆盖两个场景；风险表中"冬季场景无源文件可对照（中）"整行删除。全项目需要人工判断的环节收敛为：视觉校色终审 + B 类镜头验收（若采纳 P3-1，后者也大幅缩小）。
